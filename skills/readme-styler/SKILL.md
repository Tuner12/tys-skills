# README Styler

## Overview

Use this skill to create README files that look strong on GitHub and communicate clearly at a glance.

A good README is not just decorative. It should create fast orientation, show why the project matters, and make the repository feel maintained and intentional.

## Core Principle

Optimize for two things at once:

- immediate visual impact
- fast comprehension of the repository

Do not add visual decoration that makes the page harder to scan.

## Workflow

1. Identify the repository type.
   - personal repo
   - open-source library
   - application
   - internal tool
   - skill collection
   - portfolio/demo project
2. Identify what the README must accomplish.
   - explain the project
   - showcase the project
   - attract users or contributors
   - make the repository look polished
   - improve trust and perceived quality
3. Build the structure before adding styling.
4. Add visual elements that support the structure.
5. Remove clutter until the page feels intentional.

## README Architecture

Use a layered structure:

1. Hero section
   - repository name
   - one-line description
   - optional badges
   - optional centered layout
2. Quick orientation
   - what this repository is
   - who it is for
   - what makes it useful
3. Main content
   - features
   - layout
   - installation
   - usage
   - examples
   - roadmap or contribution info
4. Footer
   - credits
   - status note
   - license or maintenance note if relevant

Not every README needs every section. Keep the shape matched to the project.

## Styling Moves

Use presentation elements selectively:

- badges for stars, last commit, version, status, package manager, docs, or language
- a centered hero section for portfolio-style or polished public repos
- block quotes or GitHub callouts for notes, warnings, or quick framing
- concise tables for features or project inventory
- fenced code blocks for install and usage examples
- simple ASCII tree blocks for repository structure

Prefer coherence over quantity. Five well-chosen visual elements are better than fifteen random badges.

## Good Badge Use

Badges should support credibility or quick scanning, not decorate for decoration's sake.

Strong badge categories:

- stars
- forks
- last commit
- release/version
- repo size
- top language
- license
- CI/build status

Weak badge use:

- too many badges in different styles
- badges that repeat obvious information
- badges that look broken or irrelevant

## Tone Control

Adjust style to the project:

- portfolio repo: sharper visuals, more showcase energy
- library/tool repo: clearer install and usage sections
- personal skill repo: polished overview, repository map, featured items
- serious production repo: cleaner, calmer, less decorative

Do not make a README look flashy if the project itself is sober and technical.

## Anti-Patterns

Avoid these common failures:

- giant badge walls
- huge paragraphs at the top
- decorative sections with no substance
- emoji spam
- inconsistent heading depth
- repeated content across sections
- empty roadmap or TODO sections that weaken credibility
- overdesigned README with no clear install or usage path

## Output Standard

A strong README should:

- look good in the GitHub web UI
- make the project understandable in under thirty seconds
- feel like it belongs to the repository
- balance polish with signal
- avoid generic template energy
